# Full Redesign Plan — Papope

## Overview

Redesign the Papope Next.js app with:
- A clean, dark minimal front page with public/private section split
- A unified `/login` page gating all private sections
- Multi-user system with per-user access control stored in Vercel Postgres
- An `/admin` panel for user/access management
- Each sub-part isolated in its own folder with its own CSS
- A new **Ionickel** module (mock/placeholder page)

---

## Current State

- **Framework**: Next.js 14 App Router, TypeScript, Tailwind CSS, `next-auth` v4
- **DB**: Vercel Postgres (`@vercel/postgres`)
- **Auth**: Single hardcoded user via `APP_USERNAME` / `APP_PASSWORD` env vars
- **Sections**: Oracle (public), Games (public), Chronicle (private), Keep Pushing (private)
- **CSS**: Mix of inline styles, Tailwind, and CSS modules per section
- **Floaters/blobs**: Animated emoji decorations on the homepage (to be removed)

## Phase 1 — Database: Multi-user Schema

### New tables in `lib/db.ts`

```sql
-- Users table
CREATE TABLE IF NOT EXISTS users (
  id          SERIAL PRIMARY KEY,
  username    TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,   -- bcrypt hash
  is_admin    BOOLEAN NOT NULL DEFAULT false,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Per-user access control
CREATE TABLE IF NOT EXISTS user_access (
  id      SERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  section TEXT NOT NULL,  -- 'chronicle' | 'keeppushing' | 'ionickel'
  UNIQUE(user_id, section)
);
```

### Package addition

Add `bcryptjs` + `@types/bcryptjs` for password hashing.

## Phase 2 — Auth Rewrite

### `lib/authOptions.ts`

- Replace env-var single user with DB lookup:
  1. Find user by `username` in `users` table
  2. Compare password with `bcryptjs.compare()`
  3. On success, include `id`, `username`, `isAdmin` in JWT token
  4. Fetch user's allowed sections from `user_access`, store in session token

- Change `pages.signIn` to `/login` (unified)

- Update redirect callback: after login go to `/` (home), the front page will handle routing

### Session type augmentation (`types/next-auth.d.ts`)

Extend `Session` and `JWT` to carry `userId`, `isAdmin`, `sections: string[]`.

## Phase 5 — Access Guard (middleware or per-page)

Each private section needs to check:
1. User is authenticated
2. User has access to that section (from `session.user.sections`)

**Strategy**: create `lib/withAccess.ts` — a utility used at the top of each private page:
```ts
// server component helper
async function requireAccess(section: string) {
  const session = await getServerSession(authOptions)
  if (!session) redirect('/login')
  if (!session.user.sections.includes(section)) redirect('/')  // no access
}
```

Update existing private pages:
- `app/chronicle/page.tsx` → call `requireAccess('chronicle')` instead of redirecting to `/keeppushing/login`
- `app/keeppushing/page.tsx` → call `requireAccess('keeppushing')`
- `app/ionickel/page.tsx` → call `requireAccess('ionickel')` (new)

## Phase 6 — Admin Panel

### `app/admin/page.tsx`

Protected: only accessible if `session.user.isAdmin === true`.

**Features:**
- **User list**: table of all users (id, username, isAdmin, created_at)
- **Create user**: form with username + password + isAdmin toggle
- **Delete user**: button per user row
- **Access management**: per user, checklist of sections (`chronicle`, `keeppushing`, `ionickel`)

**UI**: clean dark table with minimal styling, own CSS module `admin.module.css`.

### API routes

```
app/api/admin/users/route.ts
  GET  → list all users (admin only)
  POST → create user (admin only)

app/api/admin/users/[id]/route.ts
  DELETE → remove user (admin only)

app/api/admin/access/route.ts
  GET  → get access for a user
  POST → set access for a user (replaces all, takes { userId, sections[] })
```

All admin routes check `isAdmin` from session before responding.

## Notes

- **First admin user**: needs to be seeded into the DB. Will provide a one-time seed script or API route that reads `APP_USERNAME`/`APP_PASSWORD` env vars to create the first admin — after that, user management is through the admin panel.
- **bcrypt**: passwords stored as bcrypt hashes (cost factor 10).
- **Session sections**: loaded at login and stored in JWT — no DB hit on every request.
- **Ionickel**: private, access-controlled, but just a placeholder page for now.
